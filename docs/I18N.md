# Internationalisation (i18n) Guide

NetTools Hub is built for an international audience. This document explains how translation works and how to add or improve a translation.

> 🇬🇧 English (default) · 🇨🇳 [简体中文](./I18N.zh.md) · 🇯🇵 [日本語](./I18N.ja.md)

---

## 1. How i18n works in this codebase

### 1.1 One file, three languages

UI strings live in **a single file**: [`src/lib/i18n.ts`](../src/lib/i18n.ts). It is a 3-column flat-key table of ~36 keys:

```ts
const translations = {
  en: {
    "site.title": "NetTools Hub",
    "nav.home": "Home",
    "taxonomy.kind": "Kind",
    "table.col.name": "Name",
    // …
  },
  zh: {
    "site.title": "NetTools Hub",
    "nav.home": "首页",
    "taxonomy.kind": "类别",
    "table.col.name": "名称",
    // …
  },
  ja: {
    "site.title": "NetTools Hub",
    "nav.home": "ホーム",
    "taxonomy.kind": "種別",
    "table.col.name": "名称",
    // …
  },
};
```

A **second** small table lives in
[`src/lib/taxonomy.ts`](../src/lib/taxonomy.ts) for the kind and
platform human-readable labels (e.g. `proxy` → "Proxy cores" /
"代理核心" / "プロキシコア"). It is split out from `i18n.ts` so
the labels travel with the taxonomy — adding a new `kind` only
edits one file.

The current language is read from the `?lang=` URL parameter at
runtime, with `localStorage` as a sticky preference and
`navigator.language` as the final fallback. **There are no
cookies.** Deep links are stable URLs — sharing `?lang=zh` works
exactly like sharing `?lang=ja`.

### 1.2 The `t()` function

```ts
import { t } from "@/lib/i18n";

t("en", "site.title");                  // "NetTools Hub"
t("zh", "nav.home");                    // "首页"
t("ja", "taxonomy.kind");               // "種別"

t("en", "taxonomy.count", { n: 12 });   // "12 entries"
t("zh", "taxonomy.count", { n: 12 });   // "12 条"
t("ja", "taxonomy.count", { n: 12 });   // "12 件"
```

The placeholder syntax is **`{name}`** (no spaces around the
braces). Missing keys fall back to English, and if English is
also missing the key itself is returned (so a typo is visible on
the page rather than crashing the render).

### 1.3 The hydration contract

Static export pre-renders every page in **English** regardless
of the URL `?lang=` — the build has no per-request context. The
real language is resolved **after** hydration by
[`useClientLang`](../src/lib/use-client-lang.ts) and mirrored into
`<html lang>` by [`set-html-lang`](../src/components/set-html-lang.tsx).
This shape prevents the React 19 #418 / #423 hydration mismatch
that the previous `useState(() => resolveInitialLang())` design
suffered from. There is a single-frame English flash for non-EN
deep links, which is the documented trade-off.

### 1.4 The `langParam` / `withLang` helpers

```ts
import { langParam, withLang } from "@/lib/i18n";

langParam("en", "/explore");          // "/explore"             (canonical)
langParam("zh", "/explore");          // "/explore?lang=zh"
withLang("en", "/explore");           // "/explore"
withLang("ja", "/explore?foo=1");     // "/explore?foo=1&lang=ja"  (no double `?`)
```

The English variant of every URL is the canonical one (no
query). The other two languages opt in via `?lang=`. The
`<link rel="alternate" hreflang>` block in
[`src/app/layout.tsx`](../src/app/layout.tsx) and the
[`sitemap.ts`](../src/app/sitemap.ts) both reflect this.

### 1.5 BCP-47 tags

The mapping from our `Lang` union to BCP-47 is a single
constant in `src/lib/i18n.ts`:

```ts
export const LANG_HTML_LANG: Record<Lang, string> = {
  en: "en",
  zh: "zh-Hans",
  ja: "ja",
};

export const LANG_OG_LOCALE: Record<Lang, string> = {
  en: "en_US",
  zh: "zh_CN",
  ja: "ja_JP",
};
```

Adding a language means editing both constants in lockstep —
the TS compiler will fail any caller that misses a row.

## 2. Adding a NEW language

> Want to add Spanish / French / Korean / Russian / Arabic / …? We'd love that. Here's how.

### Step 1 — Open the i18n file

```bash
$EDITOR src/lib/i18n.ts
```

### Step 2 — Register the language

In the `Lang` union type, add your code:

```ts
export type Lang = "en" | "zh" | "ja" | "es";   // ← add "es"
```

And in the same file, add a row to `LANG_OPTIONS`:

```ts
export const LANG_OPTIONS = [
  { value: "en", label: "EN" },
  { value: "zh", label: "中文" },
  { value: "ja", label: "日本語" },
  { value: "es", label: "Español" },   // ← add
] as const;
```

### Step 3 — Add a column to the translations table

Add a new key with the **same keys as `en`**:

```ts
const translations = {
  en: { … },
  zh: { … },
  ja: { … },
  es: {                                          // ← new
    "site.title": "NetTools Hub",
    "nav.home": "Inicio",
    "taxonomy.kind": "Tipo",
    // …
  },
};
```

> 💡 The fastest way is to copy the `en` block and translate every value. Missing keys fall back to English, so you can also open a draft PR and incrementally fill it in.

### Step 4 — Update `LANG_HTML_LANG` and `LANG_OG_LOCALE`

```ts
export const LANG_HTML_LANG: Record<Lang, string> = {
  en: "en",
  zh: "zh-Hans",
  ja: "ja",
  es: "es",   // ← add
};
export const LANG_OG_LOCALE: Record<Lang, string> = {
  en: "en_US",
  zh: "zh_CN",
  ja: "ja_JP",
  es: "es_ES",   // ← add
};
```

### Step 5 — Add a kind/platform label column

The `KIND_LABELS` and `PLATFORM_LABELS` tables in
[`src/lib/taxonomy.ts`](../src/lib/taxonomy.ts) also need an
`es` key for every row. The TS compiler will catch any miss.

### Step 6 — Add a README in your language

```bash
cp README.md README.es.md
# Translate the content
```

GitHub auto-detects `README.<lang>.md` files.

### Step 7 — Open a PR

In your PR description, mention:

- Which language you added
- That you're a native / fluent speaker
- Whether you translated the README too (recommended)

We will review and merge.

## 3. Improving an EXISTING translation

Found a typo or an awkward phrasing?

### Option A — Edit the i18n file directly

```bash
$EDITOR src/lib/i18n.ts
```

Find the key (e.g. `"table.empty"`), edit the value, commit, push, open a PR.

### Option B — Open an Issue first

If you're not sure about wording, open a [Translation issue](../../issues/new?template=translation.yml) and we can discuss.

## 4. Style guide for translations

These guidelines keep the UI consistent and the strings short enough to fit.

| Guideline | Example |
|---|---|
| Keep it **short** — UI labels are bounded by width | "Explore" / "探索" / "探索" (3 chars) |
| **Don't translate** technical terms unless you have to | `Clash`, `sing-box`, `WebSocket`, `CDN` stay as-is |
| **Match the tone** — concise, professional, friendly | Don't add emoji that don't appear in the source |
| **Preserve placeholders** — `{count}`, `{n}` must remain | `t(lang, "taxonomy.count", { n })` |
| **Mind the language** — `LANG_OPTIONS` labels are *in* the language | "中文" not "Chinese" |
| **Field-manual tone** — the new design is a "field manual", not a marketing site | Prefer direct, technical phrasings over promotional ones |

## 5. Verifying your translation

```bash
pnpm dev     # http://localhost:8080/?lang=es
```

1. Open each page: `/`, `/explore`, `/explore/k/proxy/`, `/explore/k/proxy/p/desktop/`.
2. Click the Globe icon in the top-right.
3. Pick your new language.
4. Check every label, the kind card, the project table headers,
   the breadcrumb, the footer, the 404 page, the error page, the
   "no entries match this filter" message.
5. If a string is missing, it will fall back to English and the
   key itself will not appear in the DOM (the fallback is
   transparent); the assertion is the look of the page, not the
   console.
